
# DoR e DoD

Checklists de **Definition of Ready (DoR)** e **Definition of Done (DoD)** para itens do backlog (*Work Items List*).

Cada requisito deve indicar **OE** e **CP** (ver [Solução Proposta](../solucao-proposta/)).

## DoR (Ready)

- História (user story) clara e objetiva
- Critérios de aceitação (AC) verificáveis/testáveis
- Vínculo com **OE** e **CP**
- Prioridade MoSCoW definida
- Regras de negócio definidas (missões/gamificação)
- Dependências conhecidas (técnicas e UX)
- Validação definida (quem valida e canal)

## DoD (Done)

- Implementado conforme AC
- Testes/validação de fluxo executados (quando aplicável)
- Revisado e com correções essenciais aplicadas
- Validado com a cliente (síncrono ou assíncrono)
- Rastreabilidade atualizada (backlog - história - OE/CP - AC)
- Disponível para demo/homologação na iteração

## Uso no processo

- Refinamento: aplicar DoR antes de puxar para desenvolvimento
- Revisão da iteração: aplicar DoD para encerrar o item

## Exemplo mínimo (User Story)

**ID:** US-XX  
**História:** Como estudante, quero concluir uma missão diária e ganhar XP para acompanhar meu progresso.  
**OE/CP:** OE2; CP1 + CP2  
**MoSCoW:** Must  

**AC:**
- Concluir missão com evidência válida registra conclusão e soma XP.
- Painel de progresso exibe XP e nível atualizados.

**Validação com cliente:** assíncrona (e-mail); consolidar decisão em ata.
